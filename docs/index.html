#why

macOS pops "Your system has run out of application memory" before things get critical. Windows pops "Your computer is low on memory." Linux GNOME doesn't — it just starts swapping until your machine becomes unresponsive.

If you run multiple AI coding agents (Claude Code, Cursor, Aider) plus a browser with 50 tabs and a few language servers, you'll hit the freeze regularly. The kernel's OOM killer eventually fires, but by then your session is already trashed.

ram-rescue is the missing early warning layer: a systemd-user timer that wakes every 60 seconds, reads /proc/meminfo and /proc/pressure/memory, and fires a desktop notification when memory gets tight — while you can still close things gracefully.

#what it does

A small shell-script-and-systemd-timer that does four things, in order of usefulness:

1 · Periodic memory check

Wakes every 60 seconds. Reads /proc/meminfo for the percentage of available memory, plus /proc/pressure/memory for the kernel's PSI signal (more on that below). Fires a desktop notification when either trigger goes off. Exits in ~50ms. Zero resident process; no daemon to feed.

3 · Hotkey-launched kill picker

Bind any key (defaults to Ctrl+Alt+R). Pressing it opens a checklist of all running apps; tick what to close, confirm, and SIGTERM goes out. Same app classifier as the alert, so the picker shows the same friendly names. System processes (GNOME shell, X server, systemd, shells) are filtered out so you don't accidentally end your session.

4 · OOM-killer post-mortem

If the kernel killed something while you were AFK, you find out via a separate 💀 Linux OOM-killer fired · victim: <name> notification on the next timer fire. Scans journalctl -k for kernel OOM events since the last check.

+ Analytical commands

ram-rescue why shows per-app analysis with "closing X frees Y" math. ram-rescue stats gives a 7-day history of alerts, most-frequent offenders, and outcomes. Useful for "should I buy more RAM or just close tabs?"

#how Linux memory actually works

If you've ever looked at free -h and wondered "I have 16 GB of cache and 5 GB free — why is my swap at 8 GB?" — this section is for you.

The terms

TermWhat it isWhat you can do
FreeTruly unused RAM. Modern Linux deliberately keeps this low because idle RAM is wasted RAM — anything not in use is fair game for cache.Nothing — by design.
Buff/cacheFile contents the kernel opportunistically cached. Counted as "in use" in some tools, but is instantly reclaimable — when an app asks for memory, the kernel drops cache pages and hands the RAM over.Treat as available.
UsedAnonymous pages — memory allocated by applications for their own working set (not file-backed).Close apps to reduce.
SwapPages the kernel pushed to disk to free RAM at some point in the past. The kernel chooses idle anonymous pages over hot cache pages when memory pressure rises.Adjust swappiness, manually reclaim (see below).
AvailableKernel's estimate of how much can be allocated to a new app without swapping. Roughly free + reclaimable cache. This is what ram-rescue watches.Nothing directly — it's a computed value.

Why is swap stuck high when I have free RAM?

Swap is sticky. Once a page lands in swap, it stays there until something tries to read it back. It doesn't auto-shrink when RAM frees up.

So that 8 GB of swap is probably history, not current pressure. You had a memory crunch earlier (Chrome opened 30 tabs, a Python script went runaway, an AI agent allocated a big context), the kernel pushed idle anonymous pages out to swap to free up RAM, those apps haven't touched those pages since → they're still parked there. Meanwhile cache filled the freed RAM because files keep getting accessed.

PSI: the metric the percentage misses

You can be at 30% MemAvailable and in a swap-thrashing loop where every memory access costs a disk read. The percentage looks fine; the machine feels like molasses.

PSI (Pressure Stall Information, /proc/pressure/memory, kernel ≥ 4.20) is a velocity metric. It tells you what percentage of wall time at least one task was stalled waiting on memory in the last 10 seconds. When PSI is high, you're suffering — even if MemAvailable looks OK.

ram-rescue checks both. The notification title reads "Memory pressure" instead of "Low memory" when PSI fired, so you know what triggered it.

#optimizing memory usage

Lower swappiness

The vm.swappiness kernel parameter (0–100, default 60 on most distros) controls how aggressively the kernel swaps out idle anonymous pages versus dropping cache. At the default, the kernel happily pushes idle apps to disk even when cache could be dropped instead — which is great for server throughput, less great for desktop responsiveness.

For an interactive desktop running AI workloads, drop it to 10:

# One-time (lasts until reboot):
sudo sysctl vm.swappiness=10

# Persistent across reboots:
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-swappiness.conf
sudo sysctl -p /etc/sysctl.d/99-swappiness.conf

Reclaim existing swap

If you already have several GB sitting in swap from earlier pressure, force it back into RAM:

sudo swapoff -a && sudo swapon -a

Takes 10–30 seconds. Only run when you have enough free RAM to hold the returning pages — check free -h first and make sure available ≥ swap used.

Check whether swap is the real problem

Swap usage by itself isn't bad — only active swap I/O is. vmstat shows you whether the system is actively reading from or writing to swap:

vmstat 1 5

The si (swap-in) and so (swap-out) columns are what matter. Sustained nonzero values = thrashing. Zeros = swap is just parked; ignore it.

Watch memory pressure in real time

PSI gives you a live signal of how stressed the memory subsystem actually is:

watch -n 1 cat /proc/pressure/memory

The some avg10 column is the most useful — percentage of the last 10s where any task stalled on memory. Above 10% means real pain.

Limit Chrome's memory hunger

Chrome is almost always the biggest offender. A few useful flags (set via chrome://flags or command-line):

  • Memory Saver (built-in, Chrome 108+): puts inactive tabs to sleep. chrome://settings/performance → enable Memory Saver.
  • Tab discarding: visit chrome://discards to see and manually discard tabs.
  • Extensions audit: chrome://extensions — disable what you don't actively use. Each extension is a process.

The honest take

If you're hitting the RAM wall daily, the real answer is more RAM.

ram-rescue is the warning layer; tuning swappiness and Memory Saver are damage control. If you're running 3 AI coding agents and 50 tabs on 16 GB, the system is over-committed. Going from 16 to 32 GB removes the problem; tuning makes the crashes happen less often.

#faq

Why bash and not Rust/Go?

The script already shells out to notify-send, gnome-system-monitor, ps, and reads /proc/meminfo. A Go binary buys you nothing but a build pipeline and arch-specific tarballs. Bash + systemd is the idiomatic Linux answer here; the whole thing is ~500 lines and runs in 50ms per timer fire.

Why not a resident daemon?

The point is to manage RAM, not consume it. A resident daemon — even a minimal one — is ~20–40 MB always. The timer-based design is 0 MB between invocations. Each run takes 50ms and exits.

Why not auto-kill like earlyoom?

Different layer of defense. earlyoom and nohang are great for auto-killing under extreme pressure. ram-rescue is the early warning that fires before you'd want auto-kill to activate, so you can pick what to close yourself. Run both: earlyoom as the safety net, ram-rescue as the heads-up.

Does it work on Wayland?

The notification path (D-Bus → notify-send) works identically on X11 and Wayland. The hotkey binding uses gsettings on GNOME (works on both). KDE on Wayland needs manual binding via System Settings.

Why is the kill picker dialog ~200 MB?

It uses zenity, which on Ubuntu 24 pulls in all of GTK4 — about 100 shared libraries. That's the same tier as gnome-system-monitor itself (~215 MB). It's only visible during your interaction — 0 MB between invocations. A Python+Tkinter rewrite (~20 MB visible) is tracked for a future release.

Does swap-to-SSD wear out my disk?

Modern SSDs are rated for hundreds of TB of writes; a typical desktop with 8 GB of swap activity won't put a meaningful dent in that even over years. Don't disable swap to "protect" the SSD — having no swap just makes the OOM killer fire sooner. Tune swappiness instead.

Should I just buy more RAM?

Probably yes. ram-rescue is the warning layer; tuning is damage control. If you're hitting the RAM wall daily on 16 GB while running multiple AI agents, going to 32 GB is the actual fix. The tool helps make a 16 GB machine less painful, but doesn't make 16 GB enough.

Will this work on a server / headless box?

No. It needs a desktop session (notify-send + a notification daemon). For servers, use nohang or earlyoom with their email/syslog alert features instead.

The notification doesn't appear when I run ram-rescue test.

First check notify-send "hello" works at all — if not, your notification daemon isn't running. On GNOME that's gnome-shell; on KDE it's plasmashell; on minimal WMs install dunst. Also check Focus Assist / Do Not Disturb isn't muting it.

Where are the logs? 
ram-rescue logs           # last 20 entries
journalctl --user -t ram-rescue --since today

Every alert and OOM event is logged with logger -t ram-rescue, queryable via journald.
